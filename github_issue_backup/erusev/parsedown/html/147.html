tuxpoldo commented on 2014-05-17 14:38:26

You are absolutely right. My fault.

First: the name and description of the pull request was confusing. Obviously the pull request would not change the parser to always open links by default in a new window, but only if the parser is explicitly configured to do so under certain circumstances. Second: the parser is the wrong place to implement that, since this is a specific use case derived from the application.

The current version of parsedown allows to implement this functionality by creating a derived class:

    class YParseDown extends Parsedown {
        public $internalLinkRegEx = '';

        protected function element(array $Element) {
            if ($Element['name'] == 'a' && $this->internalLinkRegEx != '') {
                if ( ! preg_match ($this->internalLinkRegEx, $Element['attributes']['href']) ) {
                    $Element['attributes']['target'] = '_blank';
                }
            }
            return Parsedown::element($Element);
        }

    }

That's the right way to achieve it. Thank you for your great work!

tuxpoldo commented on 2014-05-19 11:34:38

@apfelbox probably yes, it has some logic. The reason I implemented the feature that way was, that I have to replace my old own modified version of parsedown with a current one on my live website and I wanted a quick solution with very low potential of side effects (and the less code you write... you know).

But If you look at it from another perspective, you will realise, that also this approach is also OK:

  • identifyLink is a function that parses the input.
  • element is a function that generates the output.

My intention was to modify the output, and from this point of view, implementing it in element is not totally wrong.
