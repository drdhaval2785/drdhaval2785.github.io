Add PHPUnit as a dev dependency in Composer

hkdobrev commented on 2013-11-12 19:37:31

We may have previously discussed this, but there is still no strong argument against having PHPUnit as a dev dependency.

The relevant documentation

  • PHPUnit is required to run the tests, so according to the Composer documentation it should be there.
  • It will be installed only when running composer install in the Parsedown folder and not for users installing Parsedown as a dependency.
  • It won't slow down the Travis CI build, because it is not running composer install ATM.

I have commited the composer.lock as well, but it is optional for libraries. It is just my preference, so I could amend my commit if you prefer the opposite.

hkdobrev commented on 2013-11-22 20:10:55

I have PHP installed on my OS X using brew. I don't want PHPUnit installed globally. Using Composer I can just composer install any dependencies for Parsedown. In this case I will install only dev dependencies needed to run the tests.

Another reason is you don't need to specify in your CONTRIBUTING file to install PHPUnit in order to run the tests. You just need to mention composer install and vendor/bin/phpunit.

hkdobrev commented on 2013-11-25 07:08:06

what if users want to test using a different version of PHPUnit?

The required version of PHPUnit depends on the things you use from PHPUnit. For example data providers are introduced in version 3.2. However 3.7.* is the only stable version available in Packagist https://packagist.org/packages/phpunit/phpunit for the moment.

You could specify a wide range of versions like: >=3.7.0 and ignore composer.lock. In this case Composer would install the latest stable version of PHPUnit corresponding to the version constraint.

The word require is a scary word when you don't know these things

IMHO composer.json matters only to people using Composer and/or Packagist.
