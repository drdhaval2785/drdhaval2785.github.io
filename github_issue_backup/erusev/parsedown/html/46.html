hkdobrev commented on 2013-12-17 01:11:31

I'll share my extended opinion on this one.

@rk initially stated:

I'm not using a git submodule, so to track updates I have to check tags / releases.

Who said anything about using a git submodule? This would not be my recommendation in any case. You should use Composer as stated in the readme.

Regarding tracking tags / releases. This is a good practice. You need to check the changes between releases. But you don't need to do it constantly. You just need to check them before you are updating to a new minor or major version. Not a patch version. It's best to see your current version in your composer.lock file and compare it to the latest stable release on github. Like so: https://github.com/erusev/parsedown/compare/0.4.6...0.7.6

If there are breaking changes they should be visible there or in the release notes.

But how would you know if there is a new version? composer show erusev/parsedown is a good way.

If you want to get notified for new releases you could use VersionEye.

@ckdarby

Lets say I make a solid pull request, everything looks awesome, it is good functionality, etc but I didn't update the header version and now that pull request is going to sit there until I come back & deal with a single line change.

You should never update the version in a PR. Actually nobody should update the version in a normal commit. Version should be updated only after actually committing changes, pushing (to master), wait for the CI results and in general make sure everything is fine. Only then you could create one of those "bump version" commits and after that tag this commit.

So, you're going to have to diff from the last version to this version, get a list of all the modified files and then update those headers to the new version. You won't be able to use a find & replace across all the code because that means anyone who diff's between the versions will see every file modified.

Parsedown consists basically of one file. And even if it didn't you should have the version on one place only. There is no point of keeping version numbers in every file and updating them every time everywhere.

Lets say you just released 0.8.. A couple days later you notice that you forgot to update the version in the header of a file.

As I pointed above you shouldn't create a tag, before bumping the version. You could use a simple release script which updates the version and creates a tag.

That would mean you're now going to add a single commit with a one line modification to update that file you forgot and when I run composer update it is going to tell me that this project updated and I'm going to diff to see a single line modification.

If you've followed the steps I've described you would see this problem would not be possible.

In conclusion, IMHO having a version number in the header is not needed for this project. Indeed it will create too much hurdles for the maintainer. Even if the release script is a one-line bash file, the parser would be quite fine without it.

I just hope my thoughts would help someone.

Try keeping to the best practices:

  • Use Composer.
  • Keep an eye on the releases and the release notes.
  • Use git and tags for comparing.
  • Use VersionEye if new versions of the software are crucial to you.

Separating source code and meta is good.

A Packagist badge or a simple line saying: "Latest stable version: 0.8.5" is enough.
