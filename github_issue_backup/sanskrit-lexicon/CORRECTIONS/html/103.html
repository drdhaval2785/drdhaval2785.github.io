funderburkjim commented on 2015-04-21 19:01:30

@gasyoun

Why some are in () only? All suggestions, whether in parens or no, are words in some dictionary that are

within a certain edit distance of the word in question (samudragupwa in case 001 of pgn-fuzzyalpha.txt).
samudragupta is a word in some dictionary which has a close spelling to samudragupwa, so

sumdragupta qualifies as a suggestion. NOW imagine we have a list of all suggestions based on edit
distance criterion. This list is further analyzed into two sublists, based on the preceding and succeeding
headword (in case 001 of PGN, the preceding headword is candragupta, and the following headword is
also candragupta - unusual they are the same.) If a given suggestion is alphabetically between the
preceeding and following headword, then that suggestion goes in non-parenthesized list; if the suggestion
fails to be alphabetically between, then that suggestion goes in the parenthesized list.

Why some () are empty? There are no suggestions that are NOT in alphabetical order. E.g., in case 003
there is just one suggestion Suklakarmman and it is alphabetically between ruzwaSarmman and

suSarmman, so Suklakarmman was put outside (), and there was no other suggestion that could go in
()
Should we have such a big fuzzy distance? There is no easy answer to this question. The edit-distance
maximum edit distance parameter used in the PGN fuzzyalpha.py program was 6. This generates some
false positives (bad suggestions). I don't know if there is a 'best' edit distance threshold.


funderburkjim commented on 2015-04-21 19:29:13

There is two other details of the suggestion generation that is incorporated in fuzzyalpha.py.

Suppose that the fuzzy distance parameter is set to 6. And suppose our base word is X (like samudragupwa

is case 001 of PGN). Our initial pool of candidates for suggestions, Y, is all 300,000+ words in sanhw1.txt. 

    
* An accepted suggestion Y must have the same first letter as X  (e.g., Y must start with 's') 
    
* Once we have the set S of all words Y which have the same first letter as X and whose edit distance from 
    
   X is <=6,  then this suggestion list is further reduced to include only those Y in S which have the minimum 
    
   edit distance from X.   To elaborate:  Suppose the edit distances from X happen to be 3 for some Y in S, 
    
   4 for some other Y in S, and 6 for other Y in S.   Then the FINAL suggestion list will only include those  
    
   Y with edit distance of 3. 
    
The first condition was chosen to provide some computational efficiency. 
    
The second condition was chosen as a way to prune the list of suggestions (so that list would not be too long). 
    
These choices could be debated,  but, for our purposes in generating correction suggestions they seem to  
    
be plausible.

funderburkjim commented on 2015-04-22 19:50:37

@gasyoun Re 5). PD
I've generated pd-fuzzyalpha , 1158 cases. Review of a random selection of these makes me think that
THERE ARE ALMOST NO ERRORS. Rather, I suspect these cases are simply due to the great number
of compound words in PD that do not appear in MW. Take agnicEtanya as an example. Both agni and
cEtanya are words appearing in MW, but the compound word does not appear, and the icE sequence
appearing in this compound does not appear in any MW word.
I think a different approach should be used to examine this PD faultfinder list. The reason is that 1158 cases
is, practically speaking, a large list to check manually against scans.
One different approach would be for
one of our Sanskrit experts simply to read each word in the list and judge whether the word is obviously a
correct Sanskrit word (I suspect this will be true in 90% of the cases), or whether the word needs further
examination. Then, the words marked for further examination can be examined 'manually' by comparison
to the scanned images.
Another possible approach would be to write a program to identify the words which are (like agnicEtanya)
compounds of MW headwords. Here's how the program could work. The program could successively try
all 2-fold partitions of the word, for instance

a+gnicEtanya, ag + nicEtanya, agn+icEtanya, agni+cEtanya, agnic+Etanya ... agnicEtany + a
and, for a given partition, check if both components are MW headwords. If so, the given word would be
marked as OK. The remaining words (those not marked as OK) would need further review, either by
Sanskrit experts, or manually by comparison to scan.
Note: This mechanical partition is simple-minded, that's why it would be a good first thing to try. One weakness of the mechanical partition approach is that it does not take into account sandhi that can occur
in compounds when combining nominal stems. For instance, the word akzIRendu occurs among the

1158 PD cases. This is likely a compound of akzIRa and indu, (both MW stems) but this fact would not
be found by the mechanical partition approach.
