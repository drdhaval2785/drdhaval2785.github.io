Upgrade to XRegExp v2.0.0

slevithan commented on 2012-05-22 14:58:35

I recently upgraded XRegExp from 1.5.1 to 2.0.0. The changelog is here. It includes many fixes and other changes. Probably the most relevant changes for SyntaxHighlighter are that XRegExp no longer modifies any native methods by default, and no longer throws an error if it's (inappropriately) loaded twice in the same frame.

XRegExp 2.0.0 adds a lot of inline documentation, but the file size difference after minification and gzipping is less than a kilobyte (total size is a little under 3.5 KB). Of course, XRegExp 2.0.0 also includes a lot of new and improved functionality, which could be taken advantage of in SyntaxHighlighter's core and brushes.

In addition to upgrading XRegExp, I've reviewed all usage of XRegExp in SyntaxHighlighter. As a result, the attached commits additionally make the following changes:

  • Renamed XRegExp.js to xregexp.js. This is for consistency, since the official source file has always used lowercase.
  • Changed the require statement in shCore.js from require('XRegExp').XRegExp to require('xregexp').XRegExp. Both the local file and the project on npm use the lowercase "xregexp" (npm requires lowercase project names).
  • Removed the now-obsolete tests in 015_xregexp.html. This isn't to say that any tests would be inappropriate, but the tests that were included are no longer relevant since XRegExp 2 no longer overrides native methods by default.
  • Cleaned up all regexes that use XRegExp. I've removed superfluous escapings and flags, etc., while taking care to make no functionality changes. I've also changed all calls to the XRegExp constructor from new XRegExp to simply XRegExp. The new operator doesn't make a difference with XRegExp, since the XRegExp function returns a nonprimitive value (a RegExp object) and therefore isn't technically a constructor according to ES rules.

Note that since XRegExp no longer overrides native methods by default, it's now 100% safe to distribute XRegExp alongside SyntaxHighlighter. Previously, third party scripts could break if they relied on native bugs that XRegExp fixed.

If SyntaxHighlighter was relying on any cross-browser fixes (doubtful) that XRegExp 1.5.1 applied automatically by overriding native regex methods, it's easy to address this by just using XRegExp.exec, XRegExp.test, XRegExp.replace, and XRegExp.split, rather than using the corresponding native (and sometimes buggy) RegExp and String methods. However, I suspect this will not be an issue at all.
